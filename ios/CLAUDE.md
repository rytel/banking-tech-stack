# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
tuist generate --no-open   # regenerate the workspace after any Project.swift/Workspace.swift change
tuist build                # build every module + the app, in dependency order
tuist test                 # run all 8 test targets
tuist test <SchemeName>     # run one module's tests, e.g. tuist test CoreNetworkingTests
tuist test -- --test-targets TestTarget/TestClass/TestMethod   # run a single test method
tuist graph --format png --output-path .   # regenerate graph.png
```

`*.xcodeproj` and `*.xcworkspace` are gitignored and generated by `tuist generate`. Never
hand-edit them — change `Project.swift` / `Workspace.swift` and regenerate instead.

The backend must be running for the app to actually work end to end (login, topics, live ticker):

```bash
cd ../backend
./scripts/gen-cert.sh
go run ./cmd/server   # serves HTTPS on https://localhost:8443
```

## Architecture

Tuist-generated modular workspace. MVVM + Clean Architecture:
`View → ViewModel → UseCase → Repository (protocol)`. Swift 6.0, iOS 26.0+, no third-party
SPM dependencies. See [NOTES.md](NOTES.md) for the full reasoning behind the Tuist setup and
the decisions below, and `graph.png` for the visual module graph.

### Module graph and the one hard rule

**Features depend on Core, never on each other.** `App` is the only module allowed to depend
on more than one Feature — it composes them in `Projects/App/Sources/CompositionRoot.swift`.
This rule is enforced structurally: a Feature's `Project.swift` simply never lists another
Feature as a dependency.

| Module | Purpose |
|---|---|
| `App` | Composition root and dependency injection |
| `Features/Auth` | Login screen (`AuthView`, `AuthViewModel`, `LoginUseCase`) |
| `Features/TopicsList` | Topic list with search, plus the live ticker |
| `Features/TopicDetail` | Topic detail screen |
| `Core/Models` | Shared domain models and repository protocols |
| `Core/Networking` | HTTP client and repository implementations |
| `Core/SecureStorage` | Token and secret storage |
| `Core/DesignSystem` | Shared UI building blocks |

### Repository protocols live in `Core/Models`, not per-feature `Domain/`

Textbook Dependency Inversion would put a protocol next to the feature that uses it. But the
module rule above means `Core/Networking` (the concrete implementer) can never depend on a
Feature to see its protocol. `Core/Models` is the one module both a Feature (consumer) and
`Core/Networking` (implementer) already depend on, so that's where the contract has to live.
This is a deliberate trade-off, not an oversight — don't move protocols into feature `Domain/`
folders to "fix" it.

### Composition root pattern

`CompositionRoot.swift` is the only file allowed to import concrete `Core/Networking` types;
everything else only ever sees protocols from `Core/Models`. Adding a new feature follows the
same shape: a repository conforms to a `Core/Models` protocol, and `CompositionRoot` wires
`Repository → UseCase → ViewModel`.

### Networking layer

`HTTPClient` (`Core/Networking/Sources/Transport/`) is a thin, generic layer: it takes a
`Request<Response>` and returns a decoded `Response`, or a typed-throws `NetworkError`. Each
domain (`Auth`, `Topics`, `Ticker`) defines its own `*Endpoints` and `*ErrorMapping` next to
itself under `Core/Networking/Sources/<Domain>/`, mapping the transport-level `NetworkError`
into a public domain error (`AuthError`, `TopicsError`, `TickerError`). Transport errors never
leak past the repository. `APIEnvironment` (`.local` / `.production`) is the single source of
base URLs for both HTTP and WebSocket.

Repository protocols and use cases declare `async throws(DomainError)` (typed throws, SE-0413),
not plain `async throws` — this makes the compiler guarantee that no transport error can leak
into a feature. Keep using typed throws when adding new use cases or repositories.

### Live ticker (WebSocket)

`TickerRepository` bridges `URLSessionWebSocketTask`'s callback-based receive loop into a
Combine `AnyPublisher`. Cancelling the Combine subscription (`TickerViewModel.stop()` setting
its `AnyCancellable` to `nil`) is what actually closes the socket, via
`handleEvents(receiveCancel:)` — there is no explicit "close" call. Keep this in mind when
touching ticker lifecycle code.

### State and concurrency conventions

ViewModels are `@MainActor @Observable` (the `Observation` framework), not classic
`ObservableObject`/`@Published`. Swift 6 strict concurrency is **not** enabled yet — all
modules still build in default (Swift 5) language mode, so don't assume
`-strict-concurrency=complete` diagnostics are active.

### Testing split (deliberate, not accidental)

`Core/*` test targets use XCTest. `Features/*` and `App` use Swift Testing (`@Test`/`#expect`).
This split is intentional (kept for a planned XCTest-vs-Swift-Testing comparison) — don't unify
it. `MockURLProtocol` (`Core/Networking/Tests/Support/`) is the shared way to stub `URLSession`
responses in repository tests. Swift Testing structs that build `@MainActor`-isolated view
models must themselves be marked `@MainActor`, or the build fails.

### Known gaps (don't assume these are done)

- Tokens are not yet persisted to the Keychain — `Core/SecureStorage` is still a stub protocol
  (`SecureStorage.swift`) with no real implementation.
- No certificate pinning yet.
- No jailbreak/debugger checks yet.
